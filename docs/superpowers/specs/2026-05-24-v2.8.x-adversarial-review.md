# Adversarial Review — v2.8.x Patch Series + Sprint 9 Plan

**Date:** 2026-05-24
**Reviewer:** session-internal (Codex unavailable; full self-critique)
**Subject:** v2.8.1 → v2.8.8 + `docs/superpowers/specs/2026-05-24-sprint-9-cross-doc-entity-resolution.md`

The review pushes back where the design is weak. Sections finish with **Call**: which side wins, or a third option.

---

## A. The patch series itself

**FOR:** Each v2.8.x tag is bounded, test-paired, reversible, and addresses a specific user-visible failure. The series turned a 0/4 gold (v2.8.0 baseline) into a 4/4 gold (v2.8.6 offline, v2.8.7 + LLM-on-mocked, v2.8.8 FE-likely). That's defensible *as triage*. Sprint 9 takes 1–3 days; v2.8.x got results in hours.

**AGAINST:** The patches don't compose cleanly. v2.8.4 added relaxed-tag pool. v2.8.6 added row-marker priority. v2.8.7 added strict-tag bypass for the ambiguity gate. v2.8.7 also added a decline override that bypasses the reranker's verdict. v2.8.8 tightened the dedup page-window to 0. **Each fix bypasses an earlier safeguard.** The next bug will require either another bypass or a careful revert. The code is now a layer cake of "if A and B and not C and D, do something different" — readable but not provable.

**Worse:** The triage plan in `docs/superpowers/plans/2026-05-24-post-v2.8.5-triage.md` explicitly identified Sprint 9 as the right architectural move and chose patches anyway. We're 23 fixes deep with the original plan acknowledging this is the wrong place to dig.

**Call:** The series was right when it cleared TP-1 and FN-1, defensible when it cleared TP-2, **and over-extended at v2.8.7**. We should have stopped after v2.8.5 and started Sprint 9. v2.8.8 is fine (1-line change, clearer contract) but should be the last v2.8.x.

## B. `_string_family` regex-on-regex (v2.8.7)

**FOR:** The lenient regex is a 4-line change. It's testable. It captures "JCN" from "JCN 80E" without disturbing "LPS-RK-200SP". Pragmatic.

**AGAINST:** Family classification by regex is brittle. The fuse family taxonomy is engineering ontology, not text shape. Tomorrow's fixture has "Type T" fuses or "JJN-30" UL Class T sub-types or a vendor-specific code like "PT-200ESI". The regex either:
- (a) over-extracts a too-narrow prefix → false dedup
- (b) under-extracts and returns `_NO_FAMILY` → loses the bridge entirely (FN-1 redux)

A small LLM classifier at extraction time (`classify_equipment_kind(record) -> {fuse_class_l, fuse_rk1, transformer_liquid, ...}`) replaces the regex with a structured ontology that doesn't depend on string shape. Cost is real (~$0.001/record × ~50 records/doc) but it's deterministic-with-caching after the first run.

**Call:** Regex wins for v2.8.x. LLM classifier wins for v2.9 as part of equipment inventory builder. **Do not** add more regex patches to `_string_family` — the next failure is the signal to switch.

## C. Dedup `_PAGE_WINDOW = 0` (v2.8.8)

**FOR:** Within-doc same-value cross-page records ARE almost always different positional references. Coordination studies cite the same kVA rating in N TCC tables; each is a separate row-level record. Page-exact dedup respects that. The previous `_PAGE_WINDOW=2` was wrong — it was an over-correction for cross-doc revision-shift that doesn't belong in same-doc dedup.

**AGAINST:** Edge case: a single-doc parameter that legitimately appears on adjacent pages with the same magnitude (e.g. a one-line summary at the top of section 2.1, repeated in the summary box at the top of section 2.2). Pre-v2.8.8 these dedup'd cleanly; now they survive as duplicate records → duplicate flags downstream if the value mutated cross-doc. **Flag-level dedup (v2.8.6) catches this** only when both flags point at the same Doc B record; if doc_a has two records and doc_b has two records, you get 4 paired flags.

The deeper problem: dedup is the wrong abstraction. We're trying to remove "duplicates" because the extraction layer over-produces. The cure is to extract once per equipment-occurrence (Sprint 9 inventory builder).

**Call:** v2.8.8 is right *given* the current architecture. Sprint 9 obsoletes dedup almost entirely.

## D. Gold-oracle calibration

**FOR:** The gold YAML is the closest thing to ground truth we have. The 4 expected flags + 1 expected suppression cover the obvious failure modes. The new `tests/eval/test_gold_assertion.py` (5 offline + 3 LLM-on-mocked) locks regressions cheaply.

**AGAINST:** Gold doesn't cover the *negative space*. The p2 ∆-Y 5.75%↔2% mutation in `doc_b_90pct.pdf` is a real fixture difference but isn't in gold. Either:
- (a) The mutation tooling planted it intentionally and gold is incomplete → we should expand gold
- (b) It's a fixture artifact and we should ignore it → we should explicitly note that
Currently the system flags it (correctly per its own logic) but we treat that flag as "noise" without explicit gold guidance. That's calibration drift waiting to happen.

Also, gold doesn't enumerate what *shouldn't* surface (FP candidates). FP-1 covers `150 kVA ↔ 0.15 MVA`. What about `5.75%Z` vs `5.75% (no Z)` — should that flag as INFO or be suppressed? Right now it surfaces as INFO. Gold is silent.

**Call:** Gold YAML needs **explicit expanded scope** before Sprint 9 ships: add the p2 ∆-Y mutation if it's intended (or note explicitly that it's not); add FP-2/3/... for the common noise classes (string-format-only changes, unit-only changes). Do this BEFORE Sprint 9 wires equipment inventory or we'll be re-calibrating mid-implementation.

## E. Sprint 9 spec gaps

**FOR:** The spec correctly identifies the architectural problem (no equipment inventory), proposes the right shape (per-doc inventory + cross-doc matcher), and enumerates which v2.8.x heuristics get retired. Phased rollout protects existing gold tests.

**AGAINST — multiple holes:**

1. **`canonical_id` is hand-wavy.** "stable identifier within this doc" — how is it computed? Hash of descriptors? LLM-assigned label? Row marker? The spec doesn't say. If it's hash-of-descriptors, mutated docs (the whole point of cross-doc review) will hash differently → matcher must guess. If it's LLM-assigned, you've added another LLM round-trip + cache concern.

2. **Matcher confidence is binary in practice.** Jaccard ≥ 0.8 → high; ≥ 0.5 → medium; else low. But the real cases are: descriptors `{"1000KVA", "XFMR", "liquid"}` vs `{"1000kVA", "XFMR"}` — Jaccard = 2/3 ≈ 0.67. Medium. What does the pipeline do with medium-confidence matches? Spec doesn't say. Demo failure mode: high-confidence equipment match emits TP-3 flag; medium-confidence match emits a different, less-decisive flag. UI behavior undefined.

3. **No story for new-equipment-in-B.** Spec mentions `equipment_added` as "lower severity" but the gold YAML has no equivalent expectation. Reviewers expect to see additions for sign-off. This needs a UI lane.

4. **Cost model missing.** Spec hand-waves "extra LLM pass". On the locked fixture (2 docs, ~9 pages each, ~50 records per doc), inventory builder + cross-doc matcher = how many calls? At what total $? Without this the budget conversation never happens.

5. **Vision-lane integration is "fallback"-shaped.** Spec says "On diagram pages, use vision lane's entity_kind+entity_id directly." But vision is stochastic (we've seen it). What's the contract when vision returns conflicting kinds for the same physical entity across runs?

6. **No retire-order proof.** Spec table says v2.8.4 relaxed-tag align pool gets retired. But the retirement order matters: if you remove relaxed-tag before inventory is solid, you lose currently-passing flags. Each retirement needs a "switch test": gold passes with the heuristic ON, then ALSO passes with it OFF (because inventory replaces it). Spec doesn't enumerate the switch tests.

**Call:** Sprint 9 spec needs a **detail pass** before kickoff. Spend a half-day expanding §3.1 (canonical_id algorithm), §3.3 (matcher confidence buckets + downstream behavior), §3.5 (added-equipment lane), §8 (cost projection), and add a §10 "retirement switch tests" section. Otherwise the implementer hits the same shape problems mid-sprint and we're back here in 3 days with v2.9.1 patches.

## F. Vision-lane hallucination guard

**FOR:** Substring-check on PyMuPDF text is a fast, cheap, deterministic guard. Without it the vision lane fabricates entities (proto 1 showed this). False positives drop ~7-20% in the field-trip runs — meaningful filtering.

**AGAINST:** False negatives. Diagrams sometimes contain text that PyMuPDF can't extract:
- Text inside embedded images (label inside a transformer symbol rendered as raster)
- Rotated text (some TCC plot axis labels)
- Post-processing annotations (rubber-stamp comments added after the original PDF)

The guard rejects legitimate vision claims for any of these. We see warnings like `hallucination guard dropped 11/28 claims` — 39% drop rate on some pages. Some fraction of those are real entities the guard discards.

**Call:** Guard is right for v2.8.x — the false-positive class it blocks is worse than the false-negative class it creates. v2.9 should add a **secondary OCR check** (vision-extract page text via a cheap OCR pass; substring check against that union'd with PyMuPDF text). Catches the rotated/embedded cases without weakening the entity_id requirement.

## G. Page-scoped checklist gap strictness

**FOR:** Catches FN-1 correctly. The reviewer-mental-model fit is right — "row removed from this table" is the audit signal, not "value appears somewhere".

**AGAINST:** Over-flags when the description column is cleared en masse (as in doc_b p7). Three "extra" gap flags for fuses that appear elsewhere in doc_b. The migration logging mitigates but doesn't suppress.

**Call:** The right fix is **row-structure-aware gap detection**, not more permissive page-scoping. If doc_a p7 has 35 rows and doc_b p7 has 35 rows but row 34 specifically is missing the entity → gap. If doc_b p7 has descriptions-column blanked for ALL rows → that's a DIFFERENT class of issue (treat as `table_truncated` or `description_column_removed`), single flag, not 35 fuse-gap flags. This needs structure extraction (Camelot or LLM-table-parse) — v2.9 scope, not v2.8.x.

## Cross-cutting

**Test calibration debt:** Almost every v2.8.x test in this series was paired with an exact pattern from the field-trip log. That's good engineering but also means the test suite tracks a particular fixture's quirks. If the demo expands beyond the locked Eaton fixture, half of these tests become noise — passing on patterns no other doc has.

**LLM-on test mocking gap:** The LLM-on gold test mocks `_call_claude_pair` with a single permissive response. Real reranker behavior is far more diverse. The mocked tests gave false confidence at v2.8.7 — they passed while FE failed.

**Sprint 9 prerequisites:** Before Sprint 9, ship **two cheap cleanups**:
1. Expand gold YAML with the explicit FP catalog (FP-2 string-format, FP-3 unit-only, FP-4 the p2 ∆-Y question)
2. Write a `run_against_fixture.py` script that runs the full pipeline with real LLM calls and writes the result JSON to disk — used as input to a "diff-vs-yesterday's-result" regression that catches FE-mode regressions without re-running the whole pipeline manually

These take ~2 hours and would have caught the v2.8.7-passing-but-FE-failing gap before v2.8.8.
