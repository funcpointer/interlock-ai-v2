## 3. Open Failure Classes

### Class A — Cross-doc same-equipment, mutated value (TP-3)

**Symptom:** Doc A p7 has `Transformer Rating = 1000 KVA` (LLM-extracted, `entity_tag="1000KVA XFMR"`); Doc B p7 has `Transformer Rating = 100 KVA` (regex-extracted, `entity_tag="1"` row marker, AND LLM-extracted with `entity_tag="100KVA XFMR"`).

**Root cause:** the tags ENCODE the value being mutated. There is no stable cross-doc handle for "row 2 transformer slot on TCC3 page". `align_exact` v2.8.4 relaxed fallback pools the candidates, but the **count-ambiguity gate** forces value-equality fallback (because n_a > 1 or n_b > 1 in the bucket) → 1000 ≠ 100 → no pair.

**Fix candidates:**
1. **Cross-doc entity resolution** (Sprint 9 P0 scope). Build a per-equipment fingerprint from descriptor tokens (e.g. `"1000KVA"` + `"XFMR"` + page neighborhood); group records across docs whose fingerprints overlap. Pair fingerprints first, values second.
2. **Tag-stem matching** in relaxed pool. Strip the leading numeric+unit from `entity_tag` ("1000KVA XFMR" → "XFMR"). Allow pairing when stems match. Risk: bridges 150KVA XFMR ↔ 100KVA XFMR (different slot) as well as 1000KVA XFMR ↔ 100KVA XFMR (correct mutation). Mitigation: emit at very low pairing_confidence (< 0.5) and trust judge.
3. **Multi-instance positional anchor.** Track 1 regex extracted on doc_a p7 had `entity_tag = "1"` / `"2"` from row markers; same for doc_b p7. Both regex sides should pair on row index. But dedup priority `vision > llm_text > regex` kept the LLM-tagged versions and dropped the regex with row markers. Refine dedup: PREFER regex when its entity_tag is a row marker AND vision/llm_text has no row marker.

### Class B — Cross-page same-anomaly duplication (#3 + #5)

**Symptom:** doc_a p6 5.75 ↔ doc_b p2 2 AND doc_a p8 5.75 ↔ doc_b p2 2. Same underlying inconsistency (doc_b's one-line annot says 2% while its TCCs say 5.75%) flagged N times.

**Root cause:** doc_a has Transformer Impedance records on p2, p3, p4, p5, p6, p7, p8 (every TCC + one-line page). Each can semantic-pair with doc_b's lone 2% record on p2. Multiple flags emit from the same physical anomaly.

**Fix candidates:**
1. **Flag-level dedup.** Post-detect, collapse flags with identical `(canonical_name, normalized_value_a, normalized_value_b)` tuples — keep ONE (highest confidence or smallest cross-page distance). Cheap, deterministic, well-bounded.
2. **Same-doc record dedup before align.** Doc A's 7 records of `Transformer Impedance = 5.75%` should collapse to one (the most authoritative). Then only one pair survives. Slightly more invasive (changes other downstream behaviors).
3. **Hybrid:** dedup at record level when raw values within same doc are equal AND same canonical name; keep one record per (doc, name, value) cluster. Cheap and orthogonal to alignment.

### Class C — Different transformers paired across docs (#4)

**Symptom:** doc_a p7 `5.75%Z (1000KVA XFMR liquid)` ↔ doc_b p7 `2.00% (150KVA XFMR dry-type)`. Same page, same parameter name, but different physical equipment (mutated description columns made the LLM bind value to wrong slot).

**Root cause:** doc_b p7's table descriptions column is GONE (text mutation removed it). LLM saw the "2.00% Dry Type" floating text and attached it to whatever entity it could find. Cross-doc the only handle is "Transformer Impedance on p7" → pair against doc_a's two records (1000kVA + 150kVA). Reranker picks one arbitrarily.

**Fix candidates:**
1. **Same as Class A fix #1** — cross-doc entity fingerprint. The 2.00% on doc_b p7 fingerprints as 150KVA-context (`"Dry Type"` keyword), pairing it correctly with doc_a's 150KVA row → values match (2.00% vs 2.00%) → no flag.
2. **Reranker keyword extraction.** Have reranker explicitly extract equipment-context tokens (1000kVA, 150kVA, liquid, dry-type) and refuse pair when contexts differ. Heuristic; doesn't generalize.

### Class D — Checklist gaps (FN-1)

**Symptom:** Doc A p7 has `LPN-RK-500SP` (row 34); doc_b p7 has the line removed. Should surface as a missing-from-B gap with `authority_rule="checklist_gap"`. Currently zero gap flags.

**Root cause:** The detector exists (`detect/checklist.py` shipped in v2.8.4) and is wired in `pipeline.py`. But on this run, `LPN-RK-500SP` appears in `Fuse Designation` records for **both** docs (regex tally shows `[p2 regex] 'LPN-RK-500SP'` on doc_b — the one-line still has the label). So `b_values` set contains `"LPN-RK-500SP"` → gap not emitted.

This is actually correct behavior: the fuse model number appears SOMEWHERE in doc_b, so it's not literally missing. The gold's framing ("LPN-RK-500SP row removed from TCC3 table on p7") is a finer-grained gap than the current detector resolves.

**Fix candidates:**
1. **Page-scoped checklist gap.** Refine detector to scope by `(doc, page)`: if value appears in A p7 but is missing on B p7 specifically, flag. Risk: false positives when value migrated to a different page in B (typical revision flow).
2. **Table-row-context gap.** Compare doc_a vs doc_b table-row presence on the same page (requires Camelot table extraction). Higher precision; more scope.
3. **Accept as out-of-scope for v2.8.x** — current detector covers the "value entirely missing from B" case, gold FN-1 specifically is the harder "value migrated/removed from one page" case. Defer to Sprint 9.

### Class E — Noise reduction (parameter-name proliferation)

**Symptom:** Vision + LLM emit ~80+ distinct `name` values per doc (`Cable Run`, `Cable Specification`, `Cable Damage Threshold Label`, etc). Most never pair cross-doc. The alias map v2.8.4 collapsed some; many remain.

**Fix candidate:** Continue expanding `_PARAM_NAME_ALIASES` as new variants appear. Or run an LLM canonicalizer at extract time.

Reverses the current `vision > llm_text > regex` priority for this specific case.
